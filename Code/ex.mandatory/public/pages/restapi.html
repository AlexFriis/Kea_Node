Rest API

REST

Important to note that REST is a set of architectural constraints and not a protocol or a standard.
These constraints secure a better overview, better maintainabilily and longevity.
However there are still conventions and principles we have to follow.

In order for an API to be considered restful it has to conform to these criteria:
A client-server architecture made up of clients and servers, with requests managed through HTTP.
A uniform interface been components so that information is transferred in a standard form.
A client-server design pattern that enforces the seperation of concerns. To help client and server evolve independently.
Stateless - Which mandates that each request from the client to the server must contain all necessary information to complete requests.
Cacheable - requires that a response should be implicitly or explicitly label itself as either cacheable or non-cacheable.
A layered system that allows an architecture to be composed of hierachical layers by constraining component behaviour.
Code on demand (Optional) - Which allows client functionality to extend by downloading or executing code in the form of applets or scripts.

Naming Conventions

In REST all primary data representation is called a resource. Having a consistent naming strategy is one of the best design decisions in the long term.
the RESTful URI should refer to a resource that is a thing rather than an action - therefore we use nouns.
The document resource is a singular concept therefore we apply the same principle and denote document resources as singular.
We use the plural name when denoting the collection resource archetype.
A store is a client-managed resource collection therefore we use plural to denote the store resource archetype.
As controller resources are actions we use verbs when naming them - ../playlist/play.
Consistency is key - keeping maintainability and readability in focus.

Few examples:
REQUEST - Endpoint - action
GET - /movies - Retrieves all Movies
GET - /movies/{id} - Retrieves a single resources by respective ID
POST - /movies - Creates a new resource
PUT - /movies/{id} - Updates an existing resource by ID
DELETE - /movies/{id} - Deletes an existing resources by ID
